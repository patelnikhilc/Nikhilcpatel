Azure Compute Extensions: How Do You Build a Platform That Runs on Millions of VMs?

The Problem

Azure customers couldn't extend VMs after deployment. Want to install a monitoring agent? Manual SSH into each VM. Need to push a security compliance tool? Write brittle scripts that fail silently on a few machines. Forget about it scaling—at 100K VMs, you're blind to which ones actually got the configuration applied.

Microsoft's internal teams were suffering too. Security was building their own agent infrastructure. Monitoring was building theirs. Compliance was building theirs. Each team's solution was redundant, incompatible, and fragmented. There was no unified way to extend a VM at scale.

The opportunity was clear: a unified extensibility platform with network effects. More publishers bring more extensions. More extensions attract more customers. More customers attract more publishers. But networks only thrive when you solve the hard problem: governance. Without it, quality collapses, trust erodes, and the platform dies.

The Insight

"The platform isn't the product—the ecosystem is."

I approached this as a network effects play from day one. The control plane, the API, the security model—all had to be designed around a central insight: the platform's value came from the network of publishers and customers, not from any single technical feature. A slick UI or clever algorithm means nothing if publishers don't want to participate or customers don't trust what they're deploying.

That meant building governance into the DNA. Early on, we onboarded a few low-quality publishers to hit adoption metrics faster. It backfired. Customers started avoiding the marketplace. We reversed course, implemented certification, raised quality bars ruthlessly, and adoption reaccelerated. The lesson: governance is your competitive moat. It's what separates a valuable platform from a dumping ground.

How It Works

Imagine a simple flow: A publisher builds an extension (a compliance scanner, monitoring agent, whatever). They submit it to the Compute Gallery. Azure certifies it. An enterprise customer says "I want this extension on all my VMs in all regions." One API call. The platform handles the rest: downloading the extension to each VM via a lightweight agent, installing it, ensuring it's running, reporting health back to the customer, rolling back if something goes wrong.

The simple version is: Publishers → Control Plane → Orchestration → Target VMs → Observability. The control plane holds the manifest. The orchestration engine ensures every VM matches desired state. Observability tells you what's actually happening.

The Hard Choices

Every platform decision shaped adoption. Here are the four choices that defined the platform.

Declarative, Not Imperative

At 100K VMs, scripts fail silently. You don't know which machines got the update. Desired-state reconciliation ensures consistency: "all VMs should have extension X version 1.2.0." If it fails on VM 47, the system retries. You always know the actual state.

Read the architecture →

Curated, Not Open Marketplace

We could have allowed any publisher to submit extensions. We tried it. Customers flooded us with complaints about low-quality extensions. We flipped the model: certification required. Quality over quantity. Trust rebuilt adoption.

Read the architecture →

Bundled Pricing, Not Per-Extension

Per-extension fees create friction. Customers hesitate to pilot. Bundled pricing (included with compute) removes barriers. Pilot at zero cost. See value. Scale. This GTM decision probably accelerated adoption by 3-4x.

Read the architecture →

Cross-Cloud via Arc

Enterprise customers run hybrid: Azure, on-prem, AWS, GCP. One control plane for all of them. Azure Arc (outbound-only connections) made it possible. Same API, multiple clouds. This unblocked a multi-billion-dollar TAM.

The Impact

Revenue: $65M in Year One. Extensions became a key lever for Azure compute consumption and stickiness.

Strategic: Competitive moat vs. AWS Systems Manager. Network effects meant the longer the platform ran, the harder it was to compete against.

Customer: Configuration that used to take hours (manual per-VM) now takes minutes. Visibility replaced blindness. Compliance teams could enforce policies at scale.

Ecosystem: Enabled 100+ ISV partners to distribute to millions of VMs without building GTM. Created jobs. Built a thriving marketplace.

What I'd Tell an Interviewer

These are the lessons I'd walk into your office and talk through over coffee.

Governance Is Your Competitive Advantage

Most teams think governance is overhead. It's the opposite. In a network effects platform, governance prevents a tragedy of the commons. Low-quality publishers destroy trust faster than quality ones build it. Invest in certification, enforcement, and community norms. It's the moat.

Pricing Shapes Everything

Pricing isn't a revenue lever—it's a product decision. Per-feature pricing makes customers estimate ROI before adopting. Bundled pricing removes friction. Tiering signals which customers you're chasing. We tripled adoption when we shifted our pricing model. Pricing shapes adoption velocity as much as product quality does.

Backward Compatibility Is a Moat

At scale, you have dozens of extension versions live in 60+ regions on different VM agent versions. Breaking compatibility kills trust. Strict semantic versioning, compatibility matrices, and auto-upgrade paths gave customers confidence. That confidence drove adoption. This lesson applies everywhere: if your customers can't upgrade safely, they won't.

Data Drives GTM Strategy

Which extensions have highest adoption? Which verticals are fastest-growing? Which customer segments need support? That data shaped everything: which ISVs to recruit, which features to build, which customers to prioritize. Every go-to-market decision was informed by telemetry. Instinct fails at scale. Data doesn't.
