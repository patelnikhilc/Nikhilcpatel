The Challenge

When I joined Citi's Mortgage Division as VP of Application Development in July 2012, the bank's loan origination system was a legacy monolith struggling under the weight of volume, regulatory complexity, and competitive pressure. The system's architecture couldn't scale to meet demand—processing took 10+ hours, and compliance deadlines were looming.

The regulatory landscape was shifting dramatically. The CFPB's TILA-RESPA integrated disclosure rule created hard deadlines that older systems simply couldn't meet. Millions of mortgage transactions were crawling through batch processes overnight. Loan closings took 40+ days—uncompetitive in a market where speed meant market share. The infrastructure was fragile, with 24/7 production support consuming enormous resources fighting fires.

The fundamental challenge: How do you modernize a mission-critical system serving millions of customers without disrupting active operations? How do you introduce regulatory compliance at scale? And how do you do it all within a $45M budget while managing a 60+ person cross-functional team?

Technical Architecture & Design

The core challenge wasn't just modernization—it was re-architecting a system handling $B+ in monthly mortgage originations without downtime. The solution: a six-layer SOA platform reference architecture built on a foundation of enterprise integration middleware, rules-driven regulatory compliance, and event-driven asynchronous processing.

Layer Deep-Dive: Channel Integration → Services Orchestration

Layer 1: Channel Integration

Branch POS Systems: Loan officers in 2,500+ Citi branches used Encompass (Ellie Mae's origination platform) to capture application data. Tibco ESB normalized Encompass XML into canonical MISMO schema.

Online Portal: J2EE web application running on WebSphere. Customers applied for mortgages directly. HTTPS/SOAP calls to ESB. API Gateway pattern: all external requests proxied through F5 BIG-IP for SSL termination and WAF protection.

Call Center: CTI (Computer Telephony Integration) with Avaya system. Loan officers accessed customer records in real-time while on calls. Integration point: call duration events triggered document assembly workflows.

Third-Party Originators (TPO): B2B gateway for mortgage brokers, loan correspondents. Batch EDI/XML file transmission over SFTP for high-volume originators. Real-time REST API for modern TPOs.

Mobile: Pre-native era: responsive web design. Later: native iOS/Android apps. All backend calls went through the same ESB gateway, ensuring canonical data model everywhere.

Layer 2: API Gateway & Protocol Normalization

Tibco BusinessWorks as ESB: Central mediation hub. Handled 40,000+ requests/day at peak volume. Every channel (branch, online, call center, TPO, mobile) spoke different protocols:

  • Encompass → SOAP/XML
  • Online Portal → HTTPS/REST (converted to SOAP internally)
  • TPO Batch → EDI/XML (converted to SOAP)
  • Call Center → Proprietary CTI format (normalized)

Content-Based Routing: ESB examined message headers, routed loan applications to appropriate services based on product type (30-year fixed, ARM, HELOC), credit profile, TPO vs. retail. Reduced downstream service load by pre-filtering.

Layer 3: Process Orchestration & Decision Engine

Tibco iProcess (BPM): Modeled the full loan lifecycle as a BPMN 2.0 workflow. Seven major states:

  • Application: Customer data captured, basic validation
  • Processing: Credit pulled, title search ordered, appraisal scheduled
  • Underwriting: Conditions issued, documentation reviewed, credit decision rendered
  • Approval: Rate locked, final underwriting approval
  • Closing: Disclosures signed, final walkthrough
  • Funding: Wire transferred to title company
  • Post-Close: Investor delivery, archive

JRules Business Rules Engine: IBM ILOG JRules hosted 2,000+ rules for eligibility, pricing, compliance, and fraud detection. Business analysts (not developers) maintained rules via rule authoring UI. Rules covered:

  • Eligibility Rules: Debt-to-income limits, credit score minimums, loan-to-value thresholds
  • Pricing Rules: Adjust APR based on credit score, loan amount, loan type, ARM schedule
  • Compliance Rules: TILA-RESPA disclosure requirements, qualified mortgage (QM) determination, ability-to-repay documentation
  • Fraud Detection: Synthetic fraud detection, income misrepresentation, property flipping patterns

Tibco EMS (Enterprise Message Service): JMS-compliant message broker. Pub/sub topics for event distribution:

  • loan.submitted — Application complete, triggers credit pull
  • underwriting.complete — Conditions satisfied, ready for approval
  • compliance.check.passed — CFPB rules validated, HMDA reportable
  • closing.scheduled — Triggers disclosure generation, attorney notification

Durable subscriptions ensured no events were lost. Even if a subscriber service was down, messages queued until processing resumed. Critical for compliance: every loan event must be auditable.

Layer 4: Core Microservices

Loan Origination Service: SOAP web service (WSDL contract). Managed loan state machine. Every state transition persisted to Oracle DB with audit trail. Received canonical loan data from ESB, invoked downstream services, published state-change events to EMS.

Credit Decision Engine: Integrated with three bureaus (Experian, TransUnion, Equifax) via secure XML gateways. Pulled tri-merge credit report, merged scores using Citi's proprietary scoring model, applied bureau overlays (fraud alerts, bankruptcy flags). Cached reports for 24 hours to avoid duplicate pulls during resubmissions.

Document Management (FileNet P8): Enterprise content management system. Stored all loan documents (1099s, pay stubs, bank statements, appraisals, title reports). Automated document classification: OCR scanned documents, extracted key fields (loan amount, dates, parties), indexed for rapid retrieval. Underwriters accessed documents through unified portal.

Pricing Engine: Real-time rate lock service. Integrated with Citi's hedging desk to get market rates. Calculated customer's APR based on: current market rate + credit score adjustments + ARM schedule + loan-to-value adjustments. Locked rates in Fannie Mae/Freddie Mac investor systems to guarantee execution.

Disclosure Generation (TRID): Automated TILA-RESPA Integrated Disclosures. Loan Estimate (LE) generated at application submission. Closing Disclosure (CD) generated 3 days before closing (CFPB requirement). Both outputs as PDF, stored in FileNet, emailed to customer and attorney.

Compliance Engine: Enforced CFPB TILA-RESPA rules, calculated HMDA reportable fields, performed flood determination (SFHA lookup), ran OFAC screening against terrorist watch lists. Generated compliance audit trail for regulatory exams.

Layer 5: Data Layer Architecture

Oracle RAC (Real Application Clusters) 11g: Two-node active-active cluster. Each node ran Oracle 11g with shared storage (SAN). Transaction load balanced across nodes. If one node failed, the other seamlessly took over. Recovery Time Objective (RTO): <5 minutes. Recovery Point Objective (RPO): 0 seconds (no data loss).

Tablespace Strategy: Partitioned loan table by origination date (monthly partitions). Three-year sliding window online (kept 36 months of data hot for customer lookups). Older data archived to cold storage. Index strategy: bitmap indexes for dimension tables (product type, state, funding source), B-tree for high-cardinality OLTP queries.

Connection Pooling: WebSphere managed data sources. 200 pooled connections per app server. Idle timeout: 5 minutes (reclaimed connections back to pool). Prevented connection leaks that would eventually exhaust DB connectivity.

Oracle Exadata: Specialized hardware + software for parallel query execution. Smart Scans: pushed predicate filtering into storage cells, returned only matching rows to database. Reduced batch processing from 10+ hours to <45 minutes. Daily HMDA reporting, portfolio risk analysis, investor settlement runs all fit within maintenance windows.

Teradata Data Warehouse: Separate OLAP environment. ETL processes (Informatica PowerCenter) moved data from operational Oracle DB to Teradata nightly. Business analysts ran complex queries on DW without impacting production OLTP performance. Portfolio analytics, origination pipeline dashboards, fraud detection models all ran on DW.

IBM DB2 Read-Only Integration: Legacy loan servicing database (downstream system). Citi's mortgage servicing operations continued on DB2. Loan origination system read servicing data (payoff amounts for refinance quotes) from read-only DB2 replica.

Communication Protocols & Integration Patterns

Inter-Service Communication

Channel → ESB: HTTPS/SOAP with WS-Security. X.509 certificates for mutual TLS authentication. Message-level encryption with WS-Security headers.

ESB ↔ Services (Internal): SOAP/WSDL over HTTP (internal network, no external exposure). JMS for async event distribution (Tibco EMS topics/queues).

Event Distribution: Tibco EMS with durable subscriptions. Services subscribed to relevant topics. If a service crashed, EMS buffered events. Service recovered → EMS redelivered buffered events → no message loss.

External APIs: HTTPS/REST for credit bureaus (modern APIs). SFTP for investor delivery (Fannie Mae batch feeds). MQ for Fed Wire integration (mortgage wire payments).

Database Connectivity: Oracle Net (TNS) with connection pooling. JMX metrics exposed application-level stats (request latency, error rates, thread counts) to monitoring systems.

Key Technical Decisions & Thought Process

1. Strangler Fig Migration: Why Not Big-Bang Rewrite?

Citi processes $B+ in mortgages monthly. Can't shut down origination. Strangler fig pattern: wrap legacy system with ESB. Incrementally redirect traffic to new SOA components. Each migration is atomic and reversible.

Result: Branch offices kept using Encompass seamlessly. Migrated 30% of loan flow to new Credit Decision Engine while legacy credit system handled 70%. As confidence grew, shifted traffic to new system. Zero production outages during multi-year migration.

2. COTS Rules Engine vs. Custom Code

Regulatory rules change quarterly (sometimes monthly). Custom code = code-test-deploy cycle per change. JRules (COTS): business analysts update rules in hours via authoring UI. Deploy same-day without touching application code.

Trade-off Analysis: JRules license cost (~$2M+ annually) vs. 10x faster regulatory response. When CFPB changed TILA-RESPA rules mid-year, JRules customers deployed fixes in 48 hours. Competitors using hardcoded rules took weeks. We won market share.

3. Synchronous Batch → Asynchronous Event-Driven

Legacy system: synchronous batch (10+ hour nightly window). New architecture: event-driven async. Loan application triggers cascading events (credit check, pricing, compliance) executing in parallel.

Result: Sub-1-hour end-to-end processing. Credit bureaus, pricing engine, document systems all invoke in parallel (event distribution via EMS). Final state aggregated when all events complete. Loan officers got results in real-time. Closings accelerated from 40+ days to 30 days.

4. Canonical Data Model (MISMO)

15+ systems with different data formats. Without canonical model: N×N integrations (O(n²) complexity). With MISMO-based canonical model: N×1 integrations (O(n)).

Impact: Every service speaks MISMO through the ESB. Encompass sends Encompass XML → ESB converts to MISMO → Credit Engine reads MISMO. Adding a new channel? Just implement one MISMO mapping, reuse all downstream services. Reduced integration effort by ~70%.

5. Circuit Breaker Pattern for Credit Bureau Integration

Credit bureau APIs have 99.5% availability. At Citi's volume (10K+ credit pulls/day), that equals ~50 failures/day. Without circuit breaker: cascading timeouts kill the loan pipeline.

Solution: Hystrix-inspired circuit breaker. If credit bureau rejects 5 consecutive requests, circuit opens. Downstream requests fail fast (don't queue). Cached credit scores from previous 24 hours used for pricing (degraded mode). Async retry queue resends pulls when bureau recovers. Circuit closes once recovery detected.

Deployment Architecture

Enterprise Infrastructure & High Availability

Data Center Topology

Primary: Citi's private data center in Rutherford, NJ. Active-active configuration. All traffic routes here.

Disaster Recovery: Dallas, TX data center. Active-passive standby. Oracle Data Guard replicates database changes synchronously (RPO: 0 - zero data loss). Failover time: <4 hours (RTO).

Network: Citi internal MPLS WAN connects data centers. Private circuits for Fed Wire connectivity. DMZ for external-facing services (credit bureaus, investors).

Application Deployment & Scaling

WebSphere Network Deployment (ND): 8 app server cluster members per environment. Deployment manager handles lifecycle (deploy, monitor, update).

Horizontal Scaling: During peak load (month-end closings, quarter-end origination surges), clone WebSphere cluster members. Add 2-4 additional servers to handle 3x nominal traffic.

Environment Pipeline: DEV (developers test locally) → SIT (system integration) → UAT (user acceptance) → STG (staging/pre-prod) → PROD. Each environment mirrors production architecture.

Release Cadence: Monthly major releases (new features). Weekly patches (regulatory, security). Hotfix capability: <4 hours for P1 production incidents.

Database Architecture

Oracle RAC 12c: 2-node active-active. 99.99% availability (target). Tablespaces partitioned by origination date. Monthly partitions, 3-year online window.

Index Strategy: Bitmap indexes for dimension tables (product type, state, approval status). B-tree for high-cardinality OLTP (loan ID, customer ID, SSN).

Connection Pooling: WebSphere managed data sources. 200 connections per app server (8 servers = 1,600 total pool). Peak volume: 95% utilization.

Batch Processing: Oracle Exadata Smart Scan parallelizes queries. Batch windows reduced from 10+ hours to <45 minutes. HMDA reporting, settlement feeds, investor delivery all fit in maintenance window.

Security & Compliance

SOX Compliance: Financial controls audit trail. Every state change logged to DB with user, timestamp, reason.

PCI-DSS: Payment processing (wire funding). Credit card data encrypted, tokenized, never stored unencrypted.

GLBA (Gramm-Leach-Bliley): Customer data privacy. Role-based access control. Loan officers see only their customers' data.

Encryption: AES-256 at rest (database TDE). TLS 1.2 in transit. WS-Security for SOAP message encryption.

Access Control: LDAP/Active Directory integration. Role-based: loan officer, underwriter, compliance, admin. Principle of least privilege enforced.

Monitoring & Operational Excellence

Splunk Dashboards: Loan pipeline health (applications submitted, in-process, completed). SLA tracking (% loans closed in <30 days). Error rates by service.

Alerting: PagerDuty integration. P1 (production down): pages on-call engineer immediately. P2 (degraded): creates ticket for next business day.

Capacity Planning: Quarterly reviews. Load testing before peak seasons (spring home-buying). Headroom analysis: can we handle 2x nominal traffic?
