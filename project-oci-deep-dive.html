The Problem

Government and defense agencies needed cloud capabilities—compute, storage, databases, networking—but couldn't use commercial cloud due to data sovereignty, security classification, and regulatory requirements. FedRAMP High certification, ITAR compliance, IL5 classification handling. These weren't nice-to-haves. They were deal blockers.

Oracle had carved out a dominant position in enterprise databases, but it was missing the government cloud market entirely. AWS GovCloud and Azure Government owned the space. We needed to build OCI National Security Region: a cloud offering that matched commercial OCI feature parity while operating in air-gapped, highly restricted environments.

The constraint wasn't just technical. It was institutional. The challenge: build a cloud in an environment where even the developers can't access the internet. No SSH into production. No remote debugging. No real-time log streaming. Every change needs cleared personnel. Every deployment needs an audit trail. Every feature needs compliance validation.

The ask was unprecedented: commercial cloud velocity in a classified environment.

The Architecture

OCI ONSR wasn't a stripped-down version of commercial OCI. It was a full reimagining of the entire stack to operate within security boundaries while maintaining feature parity.

Segment 1: Technical Architecture & Design

Sovereign Cloud Reference Architecture: 7-Layer Model

The ONSR architecture is built as a rigorous 7-layer stack, where each layer solves specific security, compliance, and operational constraints while maintaining full OCI feature parity.

Layer 1: Physical Security Boundary

The foundation is hardware-grade isolation. ONSR regions operate in SCIF (Sensitive Compartmented Information Facility) grade data centers with no internet connectivity whatsoever. Access requires top-secret/SCI (TS/SCI) clearance. Every server has a Trusted Platform Module (TPM 2.0) to detect tamper attempts. Key material is stored in Thales Luna HSMs (FIPS 140-2 Level 3), physically isolated from hypervisors. Network cards have firmware signing to prevent rogue modifications. The data center perimeter includes:

  • Air-gapped network with no external connectivity
  • Hardware Security Modules for all key management (signing keys, encryption keys, customer keys)
  • Tamper-evident hardware on every server (TPM 2.0 attestation)
  • Cleared personnel only (vetting, continuous re-clearance)
  • Physical network isolation: One-way data diodes for export, manual approval for import

Layer 2: Network Infrastructure

Within the physical boundary, we run Software-Defined Networking (OCI VCN—Virtual Cloud Network) with VXLAN overlay encapsulation. Every tenant gets their own virtual network with microsecond-level isolation. Network ACLs, NSGs (Network Security Groups), and Dynamic Routing Gateways (DRG) handle multi-tenancy isolation. FastConnect provides dedicated on-premises connectivity (no public internet). DNS is internal-only—no external recursive resolvers. All inter-region traffic (if multi-region) is encrypted via IPsec. Network topology is spine-leaf (Arista switches in commercial OCI adapted for air-gap) with:

  • OCI VCN (Virtual Cloud Network) with VXLAN overlay
  • Security Lists + Network Security Groups for micro-segmentation
  • DRG (Dynamic Routing Gateway) for inter-VCN routing
  • FastConnect for on-premises connectivity (no public internet)
  • Internal DNS resolution only (no external queries)
  • IPsec encryption for inter-region and inter-compartment traffic

Layer 3: Core Compute & Storage

Compute options include bare metal instances (customer gets full physical server with no hypervisor) and VM instances (Oracle Linux KVM hypervisor with custom kernel for security hardening). Bare metal is critical because government licenses (Oracle DB, EDA tools) are often tied to physical cores. Storage comes in three flavors:

  • Bare Metal instances: Full physical server, no hypervisor, maximum isolation for TOP SECRET workloads
  • VM instances: Oracle Linux KVM with custom SELinux kernel (DISA STIG hardened)
  • Block Volume: iSCSI-attached, AES-256 encryption at rest, replication across fault domains for HA
  • Object Storage: S3-compatible API, immutable buckets (WORM support) for compliance data
  • File Storage: NFS v3, shared access across compute instances for collaborative workloads

Layer 4: Platform Services (Feature Parity)

This layer implements commercial OCI services in the air-gap. Oracle Autonomous Database is the flagship—self-driving (ML-based tuning), self-securing, self-repairing. The challenge: Autonomous DB in commercial Oracle Cloud updates its ML models by phoning home. For ONSR, we built an offline model update pipeline: pre-train models in commercial environment, validate thoroughly, then transfer via cross-domain guard on approved media. Container Engine for Kubernetes (OKE) is restricted: no public DockerHub pulls. We maintain an internal container registry with curated, DISA STIG-hardened base images (Trivy-scanned, policy engine validated). Functions (serverless) triggers are OCI-internal only (no public HTTP). API Gateway supports rate limiting, authentication, and WAF integration. Streaming is Kafka-compatible (managed service). Data Integration service enables ETL pipelines for classified data processing.

  • Oracle Autonomous Database: ML-based tuning, offline model updates via cross-domain guard
  • Container Engine for Kubernetes (OKE): Managed K8s with restricted container registry (no public DockerHub)
  • Functions: Serverless compute, OCI-internal triggers only
  • API Gateway: Rate limiting, authentication, WAF integration
  • Streaming: Kafka-compatible managed service
  • Data Integration: ETL pipelines for classified data processing

Layer 5: Identity & Access Management

Identity is federated with DoD PKI. Users authenticate via CAC/PIV cards (Common Access Card / Personal Identity Verification)—hardware tokens issued by DoD, not passwords. OCI IAM implements compartment-based resource isolation: Tenancy → Compartments → Sub-compartments. Each compartment has its own IAM policies, quota limits, and audit logs. Dynamic Groups allow instance-level authorization (e.g., "if server has tag:project=intelligence, grant Autonomous DB read"). Service limits per compartment prevent noisy neighbor issues. All API calls are signed with SHA-256 HMAC using customer key material stored in HSMs.

  • OCI IAM with identity domains
  • SAML 2.0 federation with DoD PKI
  • CAC/PIV card authentication (hardware token-based, not password)
  • Compartment-based resource isolation (Tenancy → Compartments → Sub-compartments)
  • Dynamic Groups for instance-level authorization
  • Service limits per compartment (prevent noisy neighbor)

Layer 6: Compliance & Audit Layer

FedRAMP High maps to 325+ NIST controls. Every control has a technical implementation: encryption (SP 800-38), access logging (CISO-auditable), vulnerability scanning (STIG baseline checks). Continuous monitoring is machine-readable via OSCAL (Open Security Controls Assessment Language)—compliance data is JSON, not PDFs. The Audit Service logs all API calls immutably (365-day retention minimum). Logs are tamper-proof (chained log entries using SHA-256). Cloud Guard (threat detection engine) runs behavioral analysis—detects anomalies like "root user never logs in, but suddenly accesses sensitive data." Vulnerability Scanning Service checks container images against DISA STIG baselines. Security Zones enforce policy at compartment level—e.g., "no public subnets allowed in this compartment" is automatically validated.

  • FedRAMP High controls (325+ NIST controls mapped)
  • Continuous monitoring (OSCAL-format machine-readable compliance data)
  • Audit Service (immutable logs, 365-day retention, tamper-proof)
  • Cloud Guard (threat detection, automated remediation)
  • Vulnerability Scanning Service (DISA STIG baselines)
  • Security Zones (enforce security posture at compartment level)

Layer 7: Workload Layer

The top layer is where customers run their classified workloads. Intelligence Community agencies (NSA, NGA, CIA) run signals intelligence and geospatial processing. Defense contractors (Lockheed, Raytheon, General Dynamics) run ITAR-controlled design and manufacturing systems. Federal health systems (VA, DoD) run patient records and research. National laboratories (Oak Ridge, Sandia, LLNL) run classified physics and materials science simulations.

  • Intelligence Community workloads (SIGINT, geospatial processing)
  • Defense contractors (ITAR-controlled manufacturing, design systems)
  • Federal health systems (VA, DoD health records and research)
  • National laboratories (classified simulations and research)

Key Technical Decisions with Thought Process

Decision 1: Air-Gap vs Connected Gov Cloud

AWS GovCloud offers commercial OCI performance but with "restricted internet access" (still connects to AWS regions in US). ONSR took the opposite bet: full air-gap, zero internet. Why? Intelligence Community customers flat-out refuse any internet connectivity. TS/SCI data never touches a cable with external connectivity. Trade-off: every software update must be physically carried in on approved media ("sneakernet"). Built offline update pipeline with checksum verification and automated rollback if deployment corrupts the system.

Decision 2: Bare Metal First

Why offer bare metal in a cloud? Government workloads often have legacy licensing tied to physical cores. Oracle database licensing is per-core. Bare metal eliminates hypervisor overhead (2-5% performance penalty) and provides hardware-level isolation. A TOP SECRET workload can have its own physical CPU, no shared cache with another customer's SECRET workload. Risk: lower utilization (can't bin-pack different customer workloads on same hardware). Payoff: customers trust it enough to sign $10M deals.

Decision 3: Autonomous DB in Air-Gap

Oracle's Autonomous Database is self-tuning via machine learning. In commercial cloud, it phones home for model updates. For ONSR, we built a restricted variant: pre-train all models in commercial environment, validate extensively (performance benchmarks, regression tests), then transfer via cross-domain guard on approved media. Challenge: ML models are large (100s of MB). Solution: delta updates instead of full model replacement. Each month, new delta is validated, signed, and transferred. Fallback: if the new model degrades performance, revert to previous version automatically.

Decision 4: Container Registry Strategy

Can't pull from public DockerHub or ECR. Built internal container registry with curated, scanned base images (DISA STIG-hardened Ubuntu/CentOS/Oracle Linux). Scanning pipeline: Trivy (vulnerability scanner) + custom policy engine (rejects images with high-severity CVEs). Developers push to internal registry. OKE pulls only from internal registry (network policy enforces this). Trade-off: developers can't use arbitrary images from internet. Benefit: supply chain security—every container is vetted before deployment.

Decision 5: Feature Parity Approach

Every commercial OCI service goes through a "sovereign readiness review"—checklist of 47 criteria: Does it make external API calls? Does it phone home for telemetry? Are all dependencies air-gap compatible? Can it be deployed offline? If a service fails, it either gets a "restricted variant" or is excluded. Examples: Functions can't have HTTP triggers (no external webhooks). Streaming works (Kafka is self-contained). Email delivery service excluded (requires SMTP to external mail server). Feature velocity is slower, but customers get a cloud they can trust.

Decision 6: Cross-Domain Data Transfer

Data moves in/out of ONSR only through approved cross-domain solutions. Egress uses one-way data diodes (hardware enforced, impossible to reverse data flow). Import requires manual review (human approves before data enters air-gap). Built automated classification checking: before data egresses, system scans for "CONFIDENTIAL" markings, cryptographic keys, Personally Identifiable Information (PII). If found, blocks transfer and alerts CISO. Trade-off: friction on data movement. Benefit: prevents classified data from leaking accidentally.

Communication Protocols

  • Inter-service communication: gRPC with mTLS (mutual TLS). Certificates issued by internal CA, validated via PKI hierarchy. No public CAs.
  • API access: REST over HTTPS. Every request signed with OCI request signing (SHA-256 HMAC derived from customer key material).
  • Identity & Authentication: SAML 2.0 for user federation. CAC/PIV certificate-based auth for hardware token validation.
  • Audit & Logging: Syslog over TLS to centralized SIEM. Immutable log chain (each entry includes hash of previous entry).
  • Network encryption: VXLAN for overlay isolation. IPsec (IKEv2, ChaCha20-Poly1305) for inter-region traffic.
  • Management & Control: OCI Console (React SPA) served from internal CDN over HTTPS. No external dependencies.

Segment 2: Deployment Architecture

Sovereign Cloud Deployment Design

Deploying a cloud in an air-gapped environment requires obsessive attention to automation, offline validation, and reproducibility. A deployment to commercial OCI takes minutes. For ONSR, even with full automation, it takes weeks.

Region Architecture

Each ONSR region is a full OCI region—not a subset. Three Availability Domains (ADs) for fault tolerance. Three Fault Domains per AD. Minimum footprint: ~500 servers per region (mix of bare metal, VM hosts, storage arrays, networking). Each AD is physically separated (kilometers apart, separate power grids, separate network uplinks). Fault Domain separation is logical (different network cards, different power supplies within same data center).

Hardware Stack

Oracle-designed servers (X9-2 bare metal instances): 96 vCPUs, 2.4TB RAM, 100GbE NICs. Mellanox ConnectX-6 network cards (100Gbps, in-NIC encryption offload). NVMe SSD arrays for local storage. Thales Luna HSMs (one per Fault Domain minimum for key management). All hardware is cleared by NSF CNSS for classified use.

Software Stack

Oracle Linux 8 (DISA STIG-hardened base image). Custom kernel with mandatory access controls (SELinux enforcing mode). No unnecessary kernel modules. Every server is immutable (filesystem mounted read-only except logs). OCI control plane services deployed as systemd units (no containers for control plane). Customer workload containers run via OKE.

Network Topology

Spine-leaf architecture (Arista DCS-7280SE3 switches adapted for air-gap). VXLAN overlay (VNI per tenant, microsecond isolation). No east-west traffic between tenants (enforced by network policies). Micro-segmentation at hardware level: security groups are implemented as hardware ACLs on switch ports. FastConnect termination (if customer has on-prem connectivity) is single-homed (no backup to internet).

Deployment Pipeline

Fully offline, reproducible, auditable. Code developed in commercial OCI environment. Security review (SAST: SonarQube, DAST: OWASP ZAP). Package on approved media (encrypted USB drives, digitally signed). Physical transfer via courier to ONSR facility. Automated deployment via Terraform (custom offline provider) running on a disconnected laptop with network isolation enforced by hardware. Every deployment is recorded in audit log (who deployed, what changed, when, why). Rollback is automated (git-based version control).

Scaling Strategy

Vertical scaling: Add more servers to Fault Domain (scale up within existing region). Horizontal scaling: Deploy new ONSR in different facility (new region). Cross-region replication respects classification level (SECRET data replicates only to SECRET regions). RPO (Recovery Point Objective): 1 hour. RTO (Recovery Time Objective): 4 hours. Regular DR exercises with agency stakeholders (quarterly).

Monitoring & Observability

OCI Monitoring Service runs internally (no telemetry leaves the air-gap). Custom Grafana dashboards for ops team. Prometheus scrapes metrics from services (mTLS authenticated). Alert routing via PagerDuty equivalent (internal alerting system). Log aggregation via ELK (Elasticsearch, Logstash, Kibana) stack, all on-prem. Metrics retention: 30 days (limit storage). Logs retention: 1 year (audit compliance).

Deployment Decision Matrix

Hypervisor Selection: KVM vs VMware vs Xen

Choice: Oracle Linux KVM

KVM is Linux kernel-native (no separate management layer). VMware requires expensive licenses per core (deal-breaker for large government buys). Xen has smaller community support. KVM has Oracle's deep investment in optimization (custom kernel, CPU scheduling enhancements). Cost: $0 (open source). Hypervisor overhead: 2-3% (acceptable).

Hardware Security Module (HSM): Thales vs AWS CloudHSM vs nCipher

Choice: Thales Luna HSM

Thales Luna is FIPS 140-2 Level 3 certified (exceeds government requirements). AWS CloudHSM requires cloud connectivity (incompatible with air-gap). nCipher requires proprietary management software. Thales Luna can be deployed on-prem, air-gap compatible. Pre-existing DoD certifications reduce approval friction.

Container Runtime: Docker vs containerd vs CRI-O

Choice: containerd

Docker daemon is heavyweight (requires root, large attack surface). containerd is lightweight, K8s-native, minimal dependencies. CRI-O is newer (less battle-tested in government). containerd has been in production OKE for 3+ years.

Infrastructure as Code: Terraform vs Pulumi vs CloudFormation

Choice: Terraform

Terraform is cloud-agnostic (reduces vendor lock-in for government). Pulumi requires runtime (Python/Go/TS), larger security footprint. CloudFormation is AWS-only (incompatible). Terraform provider for OCI can run offline (validate and plan without internet). Large government organizations already know Terraform.

Compliance Framework: Manual Controls vs OSCAL Automation

Choice: OSCAL (Open Security Controls Assessment Language)

Manual FedRAMP audits take 6-12 months, require PDFs, lack reproducibility. OSCAL is machine-readable (JSON), enables continuous monitoring, automated evidence collection. Every control has a technical check (automated compliance validation). Reduces audit cycle from annual to continuous. Government is mandating OSCAL for FedRAMP by 2027.

Database Strategy: Autonomous DB vs Customer-Managed

Choice: Both

Some agencies want fully managed (Autonomous DB handles patching, tuning, backups). Others demand full control (customer-managed database, they manage availability). Offered both: Autonomous DB as managed service option (90% of customers), customer-managed for extreme control requirements (10% of customers, mostly national labs).

Design Patterns & Trade-offs

Feature Parity by Design

Every commercial OCI feature is evaluated for ONSR compatibility before release. If it can run in a classified environment, we support it. If it requires external APIs or internet connectivity, we build a restricted variant or exclude it.

Trade-off: Slower feature velocity vs. guaranteed capability for classified workloads.

Zero-Trust Architecture

No implicit trust between any components, even within the air gap. Every API call is authenticated. Every inter-service communication is encrypted. Lateral movement is impossible because there is no "inside" that's trusted.

Trade-off: Higher latency per request vs. eliminating lateral movement attack vectors.

Infrastructure as Code

Terraform-based provisioning even in restricted environments. Every deployment is reproducible, auditable, and can be validated offline before physical installation.

Trade-off: Custom provider development vs. reproducible, auditable deployments.

Compartment Isolation

Multi-tenant isolation using Oracle's compartment model with additional security hardening. Customers can share infrastructure but remain completely isolated at the workload level.

Trade-off: Reduced resource sharing efficiency vs. security assurance for classified data.

PM Insights from the Trenches

1. Constraints Drive Innovation

The air-gap requirement forced us to rethink every aspect of cloud operations. Offline package validation. Automated compliance checking. Deterministic deployments. Many of these became best practices that Oracle now applies across all regions.

2. Compliance Is a Feature

FedRAMP High wasn't a hurdle we cleared—it was the product. Customers didn't buy "cloud." They bought "cloud that solves my security problem." That reframing changed everything about how we marketed, sold, and built.

3. High-Touch Sales Math Works

At $M+ deal sizes with complex procurement, 90% POC conversion through white-glove onboarding wasn't luxury—it was ROI. One saved deal covered the cost of the entire sales team. That justified high-touch engagement all the way.

4. Feature Parity Signals Credibility

Competitors offered "sovereignty lite"—reduced features, limited service breadth. We bet that government customers wanted commercial cloud power, not a stripped-down alternative. Feature parity became our strongest competitive weapon.

5. Trust > Speed

In a classified environment, trust is the limiting factor. If customers don't believe the system is secure, no velocity matters. We chose slower release cycles with maximum transparency over fast features with questions.
