The Problem

Senior living facilities operate in a state of perpetual reactivity. Staff responds to incidents after they occur — falls, the #1 cause of injury in seniors, are rarely predicted. Documentation is manual and time-consuming, pulling caregivers away from direct care. Families struggle to stay informed about their loved ones' day-to-day well-being. Meanwhile, care teams are stretched thin, managing multiple residents with limited visibility into emerging health risks.

The opportunity is clear: build an intelligent platform that shifts care from reactive to predictive. With real-time sensor data, behavioral analysis, and AI-powered automation, facilities can prevent incidents before they happen, reduce administrative burden, and give families the transparency they deserve.

Technical Architecture & Design

EVA is built on a layered, event-driven architecture designed for real-time processing, predictive analytics, and autonomous decision-making. The system processes terabytes of sensor data monthly while maintaining sub-second latency for critical alerts. Each layer handles a distinct responsibility while maintaining loose coupling and high throughput.

Layer 1: Data Ingestion

The ingestion layer handles heterogeneous data streams from multiple sources, normalizing and timestamping each event for downstream processing.

IoT Wearables (BLE 5.0 Sensors)

  • Protocol: Bluetooth Low Energy 5.0 with extended range (~240m), used for wearable accelerometers and heart-rate monitors
  • Frequency: 50 Hz sampling rate for motion data, enabling gait analysis and fall detection within 200ms latency windows
  • Ingestion: Local BLE gateways aggregate and forward to MQTT broker; each gateway handles up to 50 concurrent wearables

Environmental Sensors (Zigbee Mesh)

  • Protocol: Zigbee 3.0 mesh networking for motion detectors, door/window contacts, and temperature/humidity sensors. Mesh topology provides reliability in buildings with radio interference
  • Coverage: Deployed in high-risk areas (bathrooms, hallways). Motion sensor events trigger contextual enrichment (time-of-day, facility occupancy)
  • Frequency: 1 Hz for environmental data (sufficient for behavioral pattern matching)

EHR Integration (HL7 FHIR R4 API)

  • Protocol: SMART on FHIR (OAuth 2.0) for secure, standards-based clinical data exchange. Supports both pull (scheduled queries) and push (event subscriptions) models
  • Data Streams: Medication lists, lab results, care plans, vital signs from hospital EMRs (Epic, Cerner)
  • Reconciliation: Medication interaction checking against clinical notes before alert generation; prevents erroneous fall-risk alerts for residents on benign prescriptions

Staff Mobile Inputs (React Native → REST API)

  • Technology: React Native mobile app for caregivers to log incidents (falls, near-falls, medication changes) in real-time
  • Integration: POST to REST API with JWT authentication; responses include server-side validation and conflict resolution if submitted offline
  • Metadata: Captures reporter ID, timestamp, free-text notes (fed to NLP pipeline for enrichment)

Layer 2: Event Streaming & Processing

High-throughput event streaming backbone handles ingestion scaling. Apache Kafka provides durability, ordering guarantees, and multi-consumer support. Flink applies complex event processing (CEP) rules in real-time.

Apache Kafka (3-Broker Cluster)

  • Topics: vitals.raw (heart rate, SpO2), motion.raw (accelerometer events), alerts.generated (AI model outputs), care-plans.updated (clinical changes)
  • Partitioning: By facility_id to maintain per-facility ordering and enable isolated scaling
  • QoS: MQTT v5 to Kafka bridge: QoS 1 (at-least-once) for vitals (acceptable loss), QoS 2 (exactly-once) for alerts and medication events
  • Retention: 7 days hot storage, archived to cloud object storage for batch retraining

Apache Flink (Complex Event Processing)

  • Processing Model: Stateful stream processing with session windows (10-second windows for fall detection, 24-hour windows for behavioral anomalies)
  • Patterns Detected: Rapid acceleration + deceleration (fall signature), prolonged immobility (person down), irregular heart rate + sudden movement (syncope risk)
  • State Management: RocksDB-backed state backend for fault tolerance; 99.95% uptime SLA with checkpoint every 5 seconds

Schema Registry (Avro Schemas)

  • Purpose: Enforces typed schema evolution for sensor data, preventing silent data corruption from schema mismatches
  • Example: vitals_event.avsc includes timestamp, resident_id, facility_id, heart_rate, spo2, quality_flags for multi-step validation

Dead Letter Queue (DLQ)

  • Purpose: Captures malformed events (validation failures, schema mismatches, upstream errors) for asynchronous analysis
  • Monitoring: DLQ depth triggers operational alerts; periodic audits investigate root causes and feed back to ingestion systems

Layer 3: AI/ML Engine

Polyglot ML stack with specialized models for different prediction tasks, deployed as containerized microservices with versioning and A/B testing support.

Fall Risk Prediction (XGBoost Ensemble)

  • Model: Gradient-boosted trees trained on 18 months of historical fall data from 120 residents (847 documented falls)
  • Features: Gait velocity (derived from accelerometer), sleep fragmentation patterns (motion sensor nighttime data), medication interaction risk score, time-of-day (falls peak 6-10am and 6-8pm), recent lab changes (creatinine, hemoglobin)
  • Output: Risk score (0-100) with 5-minute rolling window predictions; P1 alert (red) if score > 80, P2 alert (yellow) if 60-80
  • Deployment: TorchServe on FastAPI with sub-50ms latency; model versioned in MLflow, auto-serving new candidates after retraining pipeline validates on holdout test set
  • Calibration: Platt scaling to match predicted probabilities to observed fall rates; evaluated against Framingham Fall Risk Index for comparison

Behavioral Anomaly Detection (Isolation Forest)

  • Model: Unsupervised learning on daily activity pattern vectors (e.g., steps/hour, sleep duration, bathroom visits)
  • Baseline: 14-day rolling average per resident; deviations > 2 standard deviations flag anomalies
  • Use Case: Detects UTI onset (increased bathroom visits), depression (reduced activity), or cognitive decline (irregular sleep)
  • Output: Anomaly flag + contextual summary ("Activity 40% below baseline") routed to nursing staff review queue

NLP for Clinical Notes (Fine-Tuned BioClinicalBERT)

  • Foundation Model: BioClinicalBERT (pre-trained on PubMed and clinical notes), fine-tuned on 5,000 annotated facility notes
  • Task: Named Entity Recognition (NER) to extract symptoms, medications, vitals from free-text caregiver notes (e.g., "Resident complained of dizziness after morning meds")
  • Deployment: Sidecar service (FastAPI + Transformers library); 200ms latency acceptable for batch processing overnight
  • Output: Structured JSON entities fed to RAG pipeline for context-aware response generation

RAG Pipeline (Retrieval-Augmented Generation)

  • Components: LangChain orchestration → Pinecone vector database (768-dimensional embeddings from all-MiniLM-L6-v2) → LLM (Claude/GPT-4)
  • Knowledge Base: Embeddings of care protocols, resident care plans, clinical guidelines, and 90-day historical notes
  • Retrieval: Semantic search with BM25 hybrid ranking; top-3 documents provided as grounding context
  • Generation: Claude API instructed to generate SOAP (Subjective, Objective, Assessment, Plan) notes based on structured event data + retrieved context
  • Quality Assurance: Nurse review before submission; model confidence scores (via LLM logits) flag notes for human review if below 0.75 threshold

Layer 4: Agentic Orchestration (The Differentiator)

Multi-agent framework built on LangGraph enables autonomous, context-aware decision-making with human-in-the-loop oversight. Each agent has specialized tools and reasoning patterns.

LangGraph Architecture

  • Framework: LangGraph provides stateful, directed acyclic graph (DAG) execution for multi-turn agent workflows
  • State Graph: Shared state object persisted in Redis; each agent reads/writes to relevant state fields without side effects on unrelated agents
  • Advantage over AutoGen: LangGraph's graph-first approach is more interpretable and production-hardened; supports human-in-the-loop checkpoints where nurse must approve P1 alerts before escalation

Supervisor Agent (Router)

  • Role: Receives event streams, classifies event type (fall_alert, medication_change, behavioral_anomaly), routes to appropriate specialist agents
  • Logic: Rule-based classification with LLM fallback for ambiguous events
  • State Update: Appends to event_log in Redis state; broadcasts to listening agents via pub/sub

SOAP Note Agent (Documentation Automation)

  • Trigger: Receives behavioral anomaly or incident event
  • Tools: Retrieval (query Pinecone for resident context), Template rendering (structured forms for SOAP sections), Validation (flag missing required fields)
  • Process: (1) Query RAG for resident care history, (2) Extract entities from incident notes via NLP, (3) Generate SOAP using Claude, (4) Validate against facility documentation standards, (5) Route for nurse review if confidence < threshold
  • Output: SOAP note ready for EHR submission or queued for nurse revision

Alert Routing Agent (Severity-Based Distribution)

  • Trigger: Receives ML model alert (fall risk, anomaly, etc.)
  • Classification: P1 (Critical) — immediate page to on-call nurse, P2 (High) — task queue for current shift, P3 (Medium) — end-of-shift summary, P4 (Info) — dashboard only
  • Distribution: Multi-channel (SMS for P1, in-app notification for P2-3, web dashboard for P4); respects do-not-disturb schedules
  • Acknowledgment: P1 alerts require explicit nurse confirmation before closing; system escalates unacknowledged P1s after 5 minutes

Family Communication Agent (HIPAA-Safe Summaries)

  • Trigger: Daily scheduled job + ad-hoc behavioral anomalies marked "family-visible"
  • Process: Reads resident events, applies de-identification rules (remove medical jargon, convert clinical scores to plain language), generates natural-language summary
  • Output: Family portal message ("Your mother had a good week — she participated in 4 activities and moved around as usual")
  • Tool: LLM constrained by system prompt to avoid medical terminology and specific vital values without family consent

Care Plan Agent (Proactive Recommendations)

  • Trigger: Weekly trend analysis or escalation of recurring anomalies
  • Process: Analyzes fall risk trend (rising over 2 weeks?), medication interactions, behavioral patterns; queries care protocols for evidence-based interventions
  • Output: Structured recommendation ("Suggest physical therapy assessment — gait velocity down 15% this month") with supporting evidence and EHR doc reference
  • Clinical Review: All recommendations reviewed by facility care director before suggestion to attending physician

Agent Communication & State Management

  • Message Protocol: LangGraph message-passing with typed Python dataclasses (event_id, timestamp, agent_id, message_type, payload)
  • State Persistence: Redis-backed state store persists across agent invocations; agents query state to avoid redundant processing
  • Idempotency: Event IDs prevent duplicate alert generation if a message is retried; each agent checks event_log before execution
  • Scalability: Horizontal scaling via Kubernetes; each agent type runs in separate deployment with independent scaling rules (Alert Routing scales with alert volume, SOAP Note scales with batch jobs)

Layer 5: Presentation & User Interfaces

Staff Dashboard (React + Next.js 14)

  • Framework: Next.js 14 with App Router for server-side rendering and streaming
  • State Management: TanStack Query (React Query) for caching API responses; real-time updates via WebSocket subscriptions
  • Charts: Chart.js for vitals visualization (heart rate, SpO2 over 24 hours); Recharts for trend analysis
  • Role-Based Views: Nurses see priority-sorted alerts + resident detail pages, Admins see operational dashboards (system health, model drift metrics), Care Directors see weekly trend reports
  • Accessibility: WCAG 2.1 AA compliant; keyboard navigation, screen reader support for clinical workflows

WebSocket Communication (Socket.io + Redis Pub/Sub)

  • Protocol: Socket.io for real-time bidirectional communication; falls back to polling if WebSocket unavailable
  • Scaling: Redis Adapter for Socket.io enables multi-instance deployments; all instances subscribe to topic alerts:*
  • Latency: Sub-500ms delivery for P1 alerts from ML model to nurse's browser

Family Portal (PWA)

  • Technology: Progressive Web App (installable, offline-capable) for low-friction adoption
  • Features: Daily activity summary, upcoming care activities, simple messaging interface with caregivers, care plan visibility (summary level)
  • Design: Mobile-first; families typically access from smartphones during commutes

Communication Protocols

Sensor to Platform (MQTT v5 over TLS 1.3)

  • Protocol: MQTT v5.0 for lightweight IoT communication; TLS 1.3 for encryption in transit
  • Authentication: Mutual TLS with certificate pinning for each facility gateway
  • Topic Structure: facility/{facility_id}/device/{device_id}/vitals enables facility-level isolation and topic-based filtering

Service-to-Service (gRPC + REST)

  • Internal: gRPC with Protocol Buffers for high-performance internal service calls (Kafka consumer to ML model inference, Agent to Database)
  • External: REST/JSON for third-party integrations (EHR APIs, family portal frontend)

Agent-to-Agent (LangGraph Message Graph)

  • Model: Directed acyclic graph (DAG) of agent nodes; messages flow through predefined edges with type checking
  • Benefits: Prevents cycles, enables visualization of agent workflows, supports dynamic agent addition without orchestration changes

EHR Integration (HL7 FHIR R4 over HTTPS + OAuth 2.0)

  • Standard: SMART on FHIR (Substitutable Medical Applications, Reusable Technology on FHIR)
  • Authentication: OAuth 2.0 client credentials flow for backend-to-backend; Authorization Code flow for user-initiated queries
  • Endpoints: RESTful FHIR endpoints for Patient, Medication, Condition, Procedure resources

Deployment Architecture

EVA runs on cloud infrastructure optimized for healthcare workloads, combining managed services (reduced ops burden) with containerization (portability and scaling). The deployment strategy prioritizes HIPAA compliance, high availability, and cost efficiency across the pilot's 5 facilities.

Infrastructure: Azure Kubernetes Service (AKS)

Why AKS Over EKS / GKE

  • HIPAA Compliance: Azure has explicit Business Associate Agreement (BAA) coverage; EKS/GKE require third-party partnerships
  • Healthcare Integration: Native Azure IoT Hub integration for sensor device management and provisioning
  • Cost: 15-20% cheaper for healthcare workloads than GKE; commitment discounts available

Cluster Configuration

  • Topology: 3-node cluster per facility region (primary, backup, compute); auto-scaling 3-12 nodes based on load
  • Node Sizing: Standard_D4s_v3 (4 vCPU, 16 GB RAM) for general workloads; Standard_NC6s_v3 (6 vCPU, 56 GB RAM, T4 GPU) for ML inference
  • Networking: Azure Virtual Network (VNet) with NSGs (Network Security Groups) limiting ingress; Azure Private Link for PaaS service access (no internet-exposed endpoints)
  • Availability: Multi-zone deployment; 99.95% uptime SLA via Azure-managed control plane

Containerization & Orchestration

Docker Container Strategy

  • Principle: One responsibility per container; each agent, ML model, and service runs in its own container
  • Base Images: Python 3.11 slim for agents/ML services, Node.js Alpine for frontend build, nginx for reverse proxy
  • Registry: Azure Container Registry (ACR) with image scanning for vulnerabilities; immutable tags for production releases

Sidecar Pattern (Logging & Mesh)

  • Logging Sidecar: Fluentd container in every pod sends logs to Azure Monitor; centralizes troubleshooting across the fleet
  • Service Mesh: Istio for mTLS (mutual TLS) between pods, traffic splitting (canary deployments of new models), and observability (Jaeger tracing)

Helm Charts for Deployment

  • Templating: Helm charts define Kubernetes manifests for each service (Eva-streaming, eva-ml-inference, eva-agents, eva-web)
  • Customization: Per-facility values files override defaults (facility_id, alert_email, model_versions)
  • Release Management: Semantic versioning; helm upgrade supports zero-downtime deployments

ML Model Serving

TorchServe on GPU Nodes

  • Framework: PyTorch model server (TorchServe) for reproducible model deployment
  • Compute: GPU-enabled nodes (Standard_NC6s_v3 with Tesla T4 GPU) for fall risk inference; 50ms per inference supports 20 predictions/second throughput
  • Auto-scaling: Kubernetes HPA (Horizontal Pod Autoscaler) based on inference latency; scales to 10 replicas if P95 latency > 75ms

Model Registry & Versioning (Azure ML)

  • Storage: Azure ML Model Registry tracks all trained model versions with metadata (training date, validation metrics, feature versions)
  • Deployment: Automated promotion: dev environment runs latest model, staging validates against holdout test set, production deploys after 2-week validation period
  • Rollback: All model versions retained; can revert to previous version in < 5 minutes if new model degrades performance

Data Layer Architecture

Time-Series Database: Azure Cosmos DB (Time-Series Mode)

  • Why Cosmos over InfluxDB: Managed scaling eliminates ops overhead; built-in replication for disaster recovery; same-day setup vs. weeks for self-hosted InfluxDB
  • Data Model: Time-series containers with partition key /facility_id/resident_id; automatic TTL deletes data > 1 year old
  • Throughput: ~50,000 RU/s (Request Units) during peak hours, scales down to 5,000 RU/s nights; <$800/month for 5 facilities

Structured Clinical Data: PostgreSQL (Azure Database)

  • Purpose: ACID transactions for clinical records (care plans, medication lists, provider notes)
  • Configuration: Azure Database for PostgreSQL with High Availability replica in secondary region; automatic failover < 30 seconds
  • Security: Encryption at rest (AES-256) and in transit (TLS 1.3); audit logging captures all DDL/DML for compliance
  • Backup: Automated daily backups retained 35 days; weekly point-in-time restore tests verify recoverability

Vector Embeddings: Pinecone (Managed SaaS)

  • Why Managed: At pilot scale (5 facilities, ~1K residents), Pinecone's operational overhead is lower than self-hosted Weaviate. Reevaluate pgvector when scaling to 50+ facilities (cost benefit analysis favors pgvector at scale)
  • Index: 768-dimensional embeddings (from all-MiniLM-L6-v2) of care protocols, resident care plans, and 90-day historical notes
  • Indexing: Hybrid BM25 + semantic search; 200ms query latency for RAG retrieval

Agent State & Caching: Redis Enterprise

  • Purpose: High-speed state persistence for agent workflows; cache for frequently-accessed resident profiles
  • Configuration: Redis Enterprise (managed) with replication across 3 availability zones
  • Data: Serialized LangGraph state (event_log, agent_results, pending_approvals) with 24-hour TTL

Message Broker Strategy

Azure Event Hubs (Kafka-Compatible API)

  • Advantage: Eliminates Kafka cluster ops burden while maintaining Kafka producer/consumer API compatibility
  • Configuration: Standard tier (20 partitions, 1TB/day throughput) covers 5 facilities at pilot scale
  • Cost: ~$150/month vs. $800+/month for self-managed Kafka cluster (compute, networking, backups)
  • Scaling: Automatic partition scaling; no cluster rebalancing operations

Edge Computing Layer

Azure IoT Edge Runtime (Facility-Local Gateway)

  • Hardware: Intel NUC (Gen 11) per facility; runs containerized IoT Edge runtime
  • Function: Local aggregation of sensor data (BLE, Zigbee), reducing cloud round-trips. Run local anomaly detection models (Isolation Forest) before sending to cloud, detecting behavioral changes within 10 seconds
  • Offline Resilience: Local queueing if cloud connectivity lost; automatic sync when restored

Storage & Archival

Azure Blob Storage with Lifecycle Policies

  • Purpose: Document archival (SOAP notes, care plans, family communications)
  • Lifecycle: Hot tier (0-30 days, $0.0192/GB/month) → Cool tier (31-365 days, $0.01/GB/month) → Archive tier (>365 days, $0.004/GB/month)
  • Encryption: Customer-managed keys (CMK) via Azure Key Vault; automatic rotation yearly

Network Architecture

Virtual Network (VNet) & Private Endpoints

  • Isolation: Separate VNet per facility region (e.g., VNet-us-east, VNet-us-west) with no internet-routable paths
  • Private Endpoints: All PaaS services (Cosmos DB, PostgreSQL, Key Vault, Container Registry) accessed via private IP; no egress to public internet
  • Facility Connectivity: Site-to-Site VPN or ExpressRoute connects facility on-premises network to VNet; on-premises EHR systems access Cosmos DB over encrypted tunnel

Network Security Groups (NSG)

  • Rules: Inbound: SSH (bastion host only), HTTPS (load balancer), MQTT (TLS from IoT gateways). Outbound: Restricted to Azure PaaS FQDN endpoints, no general internet access
  • Logging: NSG flow logs sent to Azure Monitor for security monitoring

CI/CD Pipeline

GitHub Actions → Azure Container Registry → AKS

  • Trigger: Merge to main branch triggers multi-stage pipeline
  • Build: Docker build, image scan (Trivy), push to ACR
  • Test: Integration tests in staging AKS cluster; helm dry-run validates manifests
  • Deploy: Blue-green strategy; new deployment runs alongside old; traffic switches only after health checks pass
  • Rollback: One-command helm rollback if issues detected post-deployment

Monitoring & Observability

Azure Monitor + Prometheus + Grafana

  • Metrics: Prometheus scrapes Kubernetes metrics (CPU, memory, network) and custom app metrics (inference latency, alert latency, queue depth)
  • Visualization: Grafana dashboards for on-call engineers; threshold-based alerts (e.g., P99 inference latency > 100ms triggers PagerDuty)
  • Logging: Fluentd sidecars ship logs to Azure Log Analytics; KQL queries for troubleshooting
  • Tracing: Jaeger traces requests through service call chains; helps identify bottlenecks in agent workflows

Model Drift Detection (Evidently AI)

  • Monitoring: Weekly comparison of model predictions against actual outcomes (did the fall risk model correctly predict residents who fell?)
  • Drift Metrics: AUC, precision, recall, and calibration error; alerts if any metric drops > 5% from baseline
  • Action: Triggers retraining pipeline if drift detected; automatically deploys if new model validates on holdout set

HIPAA Compliance Architecture

Data Encryption

  • At Rest: AES-256 encryption for Cosmos DB, PostgreSQL, Blob Storage; managed by Azure Storage Service Encryption
  • In Transit: TLS 1.3 for all network traffic (MQTT, HTTPS, gRPC); mTLS for Kubernetes pod-to-pod communication
  • Key Management: Azure Key Vault stores all encryption keys; automatic key rotation yearly

Access Control & Authentication

  • Azure AD: Multi-factor authentication (MFA) for all staff portal access; role-based access control (RBAC) for API endpoints
  • Service Principals: Kubernetes service accounts with least-privilege IAM roles; agents can only read/write specific resources (e.g., SOAP Note Agent can only write to care_notes table)
  • FHIR Integration: OAuth 2.0 for EHR access; staff credentials never stored locally, tokens expire after 1 hour

Audit Logging

  • Events Logged: All user logins, data access (resident records), alerts generated, SOAP notes created/modified
  • Immutability: Logs written to Blob Storage with append-only access; 7-year retention for legal compliance
  • Monitoring: Monthly audit reports identify suspicious access patterns (e.g., staff reviewing non-assigned residents)

De-Identification & Privacy

  • Family Portal: LLM-constrained outputs remove specific vitals, medication names, clinical terminology
  • Model Training: Fall risk model trained on synthetic/aggregated data after de-identification; no identifiable information in feature vectors
  • Data Retention: Resident records deleted 90 days after discharge; backups follow same retention policy

Design Patterns & Architectural Decisions

The architecture relies on four core patterns, each chosen to solve a specific problem in real-time health monitoring.

Event-Driven Architecture

Why: Real-time sensor data can't wait for batch processing. A resident's motion sensor spike indicating a fall needs immediate action — milliseconds matter. Event-driven decouples producers (sensors) from consumers (alerts, dashboards), enabling independent scaling.

Trade-off: Higher infrastructure complexity (message broker, distributed tracing) vs. sub-second alert latency and responsiveness. Requires operational expertise in Kafka/Event Hubs monitoring.

CQRS (Command Query Responsibility Segregation)

Why: Read-heavy dashboards (family portal, staff views) need different optimization than write-heavy ingestion (sensor data flowing in constantly). Separate data models prevent lock contention and enable read replicas.

Trade-off: Dual data stores and eventual consistency challenges vs. query performance and independent scaling of reads and writes. Family portal may show data 1-2 seconds stale.

Agent-Based Architecture (Multi-Agent Orchestration)

Why: Documentation (SOAP notes), alerting (severity-based routing), and family communication each need autonomous decision-making. Agents decouple responsibilities and enable parallel processing. Agents can specialize in specific tasks (clinical notes, alert routing) without cross-contamination.

Trade-off: More complex orchestration and state management vs. modularity, extensibility, and the ability to swap agents without touching core logic. Requires careful monitoring to prevent agent deadlocks.

Pub/Sub for Alert Routing

Why: Different alert severities need different handlers. A critical fall alert pages the on-call nurse immediately. A warning about gait changes might be included in the daily shift summary. Pub/Sub decouples alert producers from consumers.

Trade-off: Eventual consistency and harder debugging vs. flexible, scalable routing without hard-coded logic. Requires idempotency guarantees to prevent duplicate alerts.
